Software Design and Architecture Unit 3

Transforming requirements into a blueprint that engineers can implement — from high-level architecture down to module-level design.

What is Software Design?

Definition (IEEE): Software design is the process of defining the architecture, components, interfaces, and other characteristics of a system or component and the result of that process.

Design is the bridge between requirements (what to build) and implementation (code). A good design makes the system correct, maintainable, extensible, and efficient. Bad design produces a system that works today but collapses under change.

The design imperative: Every software system has an architecture — even if no one designed it intentionally. Without deliberate design, architecture emerges chaotically. Deliberate design = controlled complexity.

Levels of Design

Architectural design: High-level structure — subsystems, layers, major components, communication patterns
Interface design: How components and subsystems communicate — APIs, protocols, message formats
Component design: Internal design of each component — classes, methods, algorithms, data structures
Data design: Data structures, database schema, file formats, data transformation
Mnemonic — Levels of Design: "AICD"
Architectural → Interface → Component → Data

The Design Process How design happens

Design as a Wicked Problem

Design is not a linear process — it is iterative, creative, and often wicked (each attempt to solve the problem reveals new aspects of the problem). Unlike analysis which discovers the problem, design invents the solution.

Analogy: Design is like sculpting. You start with a block (requirements) and iteratively remove and reshape until you reveal the form. You cannot design the whole thing top-down perfectly on the first pass.

The Design Process — Steps

1
Understand the problem: Study SRS deeply. Identify design constraints (performance, platform, budget). Define architectural drivers — the key decisions that will shape everything.
2
Identify design elements: What major components/subsystems are needed? Identify candidate architectures and patterns. Decompose the problem space.
3
Describe abstractions: Develop high-level models — component diagrams, collaboration diagrams. Assign responsibilities to components.
4
Evaluate and select design: Compare candidate designs against quality attributes (performance, security, maintainability). Use architectural trade-off analysis (ATAM).
5
Refine and detail: Elaborate each component's internals — class diagrams, sequence diagrams, database schema. Apply design patterns.
6
Validate design: Walk through scenarios using the design. Verify all requirements are satisfied. Check for consistency across design levels.
7
Document: Produce the Software Design Document (SDD) / Architecture Description. Create diagrams, decisions log (ADR — Architectural Decision Records).

Good Design vs Bad Design — The Test

Good design
  • Implements all requirements correctly
  • Readable and understandable
  • Provides complete picture for implementation
  • Exhibits low coupling, high cohesion
  • Extensible — new features fit naturally
  • Testable — modules can be tested in isolation
Bad design (Technical Debt)
  • Spaghetti code — no clear structure
  • God class/module — one component does everything
  • Tight coupling — changing one thing breaks everything
  • No separation of concerns
  • Hard to test (dependencies tangled)
  • Magic numbers, implicit assumptions everywhere

Software Design Principles Exam Core

Design principles are guidelines that, when followed, produce software that is easier to understand, maintain, test, and extend. They capture decades of hard-won engineering experience.

1. Abstraction

Focus on essential characteristics while hiding irrelevant details. The most fundamental design principle — without abstraction, every system degenerates into unmanageable complexity.

Procedural abstraction

A named function with a defined interface. Caller knows what it does, not how. e.g., searchBook(isbn)

Data abstraction

A class/ADT that hides its data representation. e.g., Book class — caller uses methods, not raw fields.

Architectural abstraction

Layers — each layer hides its implementation from layers above. e.g., Data Access Layer hides SQL from Business Logic Layer.

Why it works: Human working memory can hold ~7 items (Miller's Law). Abstraction lets us reason about a system at the right level of detail without drowning in everything at once.

2. Modularity

Divide the system into separate, well-defined modules that can be developed, tested, and understood independently.

Optimal module count: Cost of developing individual modules decreases as you have more modules (simpler units). But the cost of integrating them increases. The optimal number minimises total cost. Too few = complex monoliths. Too many = integration nightmare. 
LMS modules: BookCatalogue | MemberManagement | Circulation | 
             Reservations | Fines | Reporting | Authentication

5. Separation of Concerns (SoC)

A system should be organised so that each concern (feature, problem, responsibility) is addressed in a distinct place. Changes to one concern should not ripple into others.

LMS — Separation of Concerns:
  View layer:        HTML/CSS templates (rendering)
  Controller layer:  Handles HTTP requests, input validation
  Service layer:     Business logic (fine calculation, loan rules)
  Repository layer:  Database queries (SQL, ORM)
  Domain layer:      Book, Member, Loan entities (pure data + behaviour)
SoC in action: When the database changes from MySQL to PostgreSQL, only the Repository layer changes. No business logic or UI code is touched. This is the power of SoC.

7. DRY, KISS, YAGNI

DRY — Don't Repeat Yourself

Every piece of knowledge must have a single, authoritative representation. If fine calculation logic appears in 3 places, a change requires updating 3 places — and you'll miss one.

KISS — Keep It Simple, Stupid

Prefer the simplest solution that works. Complexity is the enemy of reliability. A 10-line solution you understand beats a 50-line "clever" one.

YAGNI — You Aren't Gonna Need It

Don't build features "for the future" that aren't needed now. Unused abstractions accumulate and create dead weight. Build for today's requirements.

Coupling worst → best: "CCE-CS-DM"
Content → Common → External → Control → Stamp → Data → Message
Cohesion worst → best: "CLTPCSEF"
Coincidental → Logical → Temporal → Procedural → Communicational → Sequential → Functional

Design Concepts Theory + Intuition

1. Information Hiding (Encapsulation)

Parnas (1972): Each module should hide a design decision — an aspect of the system that is likely to change. The module's interface should not expose how it makes that decision.

If a module hides how it stores books (flat file, relational DB, document store), then changing the storage mechanism only requires changing that one module — all callers remain unchanged.

// Good: Information hiding
class BookRepository {
    private db: Database; // hidden — callers don't know it's a DB
    
    findByISBN(isbn: string): Book {
        return this.db.query(`SELECT * FROM books WHERE isbn = ?`, isbn);
    }
}

// Bad: Leaking internals
class BookRepository {
    public db: Database; // caller can directly write SQL — disaster waiting to happen
}

2. Stepwise Refinement (Top-Down Decomposition)

Start with a high-level description and progressively refine each component into lower-level detail. Niklaus Wirth's approach — design in layers, each layer more concrete than the previous.

Level 1: Process loan request
Level 2: 
  1. Validate member is active
  2. Check book availability
  3. Create loan record
  4. Update book status
  5. Send confirmation
Level 3 (Validate member):
  1.1 Look up member by ID
  1.2 Check membership expiry date
  1.3 Check overdue fines threshold
  1.4 Return validation result

3. Structural Partitioning

Dividing system structure into horizontal (layers) and vertical (functions/features) partitions.

Horizontal partitioning

Divide into layers by role: UI, Business Logic, Data. Each layer serves the layer above it. Classic 3-tier architecture.

Vertical partitioning

Divide into feature-based modules: Books module, Members module, Loans module. Each module is a complete vertical slice (UI + logic + data).

Modern trend: Microservices use vertical partitioning — each service is a complete vertical slice that can be deployed independently. Monoliths typically use horizontal partitioning.

4. Refactoring

Improving internal structure of code without changing external behaviour. Essential for managing technical debt and preventing design decay.

  • When to refactor: When adding a new feature is hard due to existing structure, when code is duplicated, when a class is too large, when tests are hard to write.
  • Common refactorings: Extract Method, Extract Class, Rename, Move Method, Replace Conditional with Polymorphism, Introduce Interface.
  • Safety net: Never refactor without tests. Tests verify that behaviour is preserved.

5. Design Patterns

Reusable solutions to commonly occurring problems in software design (GoF — Gang of Four, 1994). Not copy-paste code — conceptual templates that solve recurring structural problems.

Three categories: Creational, Structural, Behavioural (covered in detail in Design Patterns tab).

6. Component-Level Design

A software component is a modular, deployable, replaceable part of a system that encapsulates its implementation and exposes a set of interfaces. Key properties:

  • Interface: The contract — what the component provides and requires
  • Encapsulation: Implementation details are hidden
  • Replaceability: Can swap implementations without changing the caller
  • Independence: Can be developed and tested independently
LMS Component: CirculationComponent
  Provides interface: ICirculationService
    + issueBook(memberId, isbn): Loan
    + returnBook(loanId): ReturnReceipt
    + renewLoan(loanId): Loan
    + getActiveLoansByMember(memberId): List<Loan>
  
  Requires interface: IBookRepository, IMemberRepository, IFineCalculator
  
  Internal classes: LoanFactory, LoanValidator, LoanPersistence

The Design Model 4 Design Dimensions

The design model is the set of design artefacts that collectively describe the complete design of a software system. It has four dimensions, each addressing different aspects of the system.

The 4-Dimension Design Model (Pressman)

Data/Class design

Transforms analysis domain objects and data relationships into class definitions and data structures. Entity classes, persistent data schema, data formats.

Architectural design

Defines overall structure — how major components are organised, how they communicate. The skeleton of the system.

Interface design

Defines how components communicate with each other, with external systems, and with users. APIs, protocols, UI design.

Component design

Detailed internal design of each component — algorithms, local data structures, interfaces between internal elements.

4. Component Design — Algorithmic Detail

// Component: FineCalculator — detailed design
class FineCalculator {
    FINE_RATE_PER_DAY = {
        STUDENT: 1.00,   // ₹1/day
        FACULTY: 0.50,   // ₹0.50/day (subsidised)
        PUBLIC:  2.00    // ₹2/day
    }
    MAX_FINE_CAP = 200.00  // Maximum ₹200 regardless of days
    GRACE_PERIOD_DAYS = 1  // 1-day grace period
    
    algorithm calculateFine(loan: Loan, memberType: MemberType):
        if loan.returnDate is null:
            daysLate = today - loan.dueDate
        else:
            daysLate = loan.returnDate - loan.dueDate
        
        daysLate = max(0, daysLate - GRACE_PERIOD_DAYS)
        
        rate = FINE_RATE_PER_DAY[memberType]
        fine = min(daysLate * rate, MAX_FINE_CAP)
        return fine

Software Architecture The System Skeleton

IEEE Definition: The fundamental organisation of a system embodied in its components, their relationships to each other and to the environment, and the principles guiding its design and evolution.

Why Architecture Matters

  • Communication: Architecture is the primary vehicle for communicating system structure to stakeholders
  • Decisions: Architectural decisions are the most consequential — they are expensive to undo
  • Quality: Architecture largely determines quality attributes (performance, security, scalability, maintainability)
  • Risk reduction: Architectural analysis early catches problems that would be catastrophic later
Architectural debt: Poor architectural decisions — a monolith when microservices were needed, synchronous coupling when async was required — become the most expensive form of technical debt. Refactoring architecture at scale costs millions.

Quality Attributes (Architectural Drivers)

Non-functional requirements that architecture must satisfy. Known as the "-ilities":

Performance

Response time, throughput, latency

Scalability

Handle increasing load gracefully

Availability

Uptime, fault tolerance, recovery

Security

Auth, encryption, audit trails

Maintainability

Ease of change, modularity

Testability

Can modules be tested in isolation?

1. Layered Architecture (N-tier) Most Common

2. Client-Server Architecture

System divided into clients (request services) and servers (provide services). Clients don't need to know implementation details of servers.

LMS Client-Server:
  Client: Browser running Angular/React SPA
  Server: Node.js/Django REST API
  Variants: 2-tier (client ↔ DB), 3-tier (client ↔ app server ↔ DB)

3. Microservices Architecture

System decomposed into small, independently deployable services. Each service: owns its data, can be deployed separately, communicates via HTTP/message queue.

LMS Microservices:
  book-service:       Catalogue, search, availability
  member-service:     Registration, authentication, profiles
  circulation-service: Issue, return, renew
  fine-service:       Fine calculation, payment
  notification-service: Email/SMS for reminders, dues
  report-service:     Analytics, management reports

Communication: REST APIs (sync) + RabbitMQ events (async)
  e.g. circulation-service publishes BookIssued event →
       fine-service and notification-service subscribe
ProsCons
Independent deployment and scalingDistributed system complexity
Technology diversity per serviceNetwork latency for inter-service calls
Fault isolation — one service fails, others continueData consistency across services is hard
Small teams own services end-to-endOperational overhead (monitoring, tracing, deployment)

4. Repository Architecture

All components share a central data repository. Components communicate by reading/writing to the repository, not directly to each other.

LMS example: All modules (Catalogue, Circulation, Fines, Reports) 
read/write to central LMS Database. Decouples modules from 
each other — they only need to know the data schema, not each other's APIs.

5. Pipe-and-Filter Architecture

Data flows through a series of processing steps (filters) connected by data channels (pipes). Each filter transforms data and passes it to the next.

LMS Report Generation Pipeline:
  RawLoanData → [Filter: Clean/Validate] → [Filter: Aggregate] → 
  [Filter: Calculate KPIs] → [Filter: Format as PDF] → Report

6. Event-Driven Architecture (EDA)

Components communicate by publishing and subscribing to events. Publisher and subscriber are decoupled — publisher doesn't know who is listening.

LMS Events:
  BookReturned → triggers: FineCalculation, BookAvailabilityUpdate, 
                           NotifyReservationQueue
  MembershipExpired → triggers: AccountSuspension, NotificationEmail
  LowInventoryDetected → triggers: PurchaseRecommendation

Architectural Genres Pressman's Classification

Definition: An architectural genre is a category of software systems that share common domain characteristics. The domain dictates which architectural styles are appropriate and what quality attributes are paramount.

Pressman identifies architectural genres as domain-driven categories that guide architectural decisions. Understanding the genre first prevents choosing an inappropriate architecture.

1. Open Systems / Network Architecture

Systems designed to operate within the internet/network infrastructure. Must handle distributed communication, interoperability, and network reliability.

  • Characteristics: Stateless protocols, REST/GraphQL APIs, CDNs, load balancers, horizontal scaling
  • Key concerns: Latency, bandwidth, security (TLS), caching, service discovery
  • Examples: E-commerce platforms, social media, web services, cloud APIs
  • Common styles: Microservices, SOA, client-server, event-driven

2. Distributed Systems

Computation is distributed across multiple machines connected by a network. Multiple processes run in parallel and communicate via messages.

  • Characteristics: No shared memory, asynchronous communication, partial failure possible
  • Key concerns: Consistency (CAP theorem), fault tolerance, consensus, distributed transactions
  • CAP Theorem: A distributed system can guarantee at most 2 of: Consistency, Availability, Partition Tolerance
  • Examples: Banking systems, airline reservations, blockchain
CAP Theorem for LMS: If the LMS goes fully distributed, which do you sacrifice? For a library, Availability + Partition Tolerance (AP) is often preferred — it's better to show a book as available (slightly stale data) than to fail completely when the network is partitioned.

3. Real-Time Systems

Must respond to events within strict time constraints. Correctness depends on both logical correctness AND timing.

  • Hard real-time: Missing deadline = system failure (airbag control, flight management)
  • Soft real-time: Missing occasional deadlines degrades quality (video streaming, trading systems)
  • Key concerns: Deterministic scheduling, interrupt handling, priority inversion, deadline monitoring
  • Common styles: Event-driven, priority-driven, time-triggered

4. Embedded Systems

Software embedded in hardware devices. Extremely resource-constrained (CPU, memory, power). Must interface directly with hardware.

  • Characteristics: Bare-metal or RTOS, device drivers, interrupt service routines, no operating system abstraction
  • Key concerns: Memory footprint, power consumption, hardware reliability, real-time behaviour
  • Examples: Medical devices, automotive ECU, industrial controllers, smart home devices
  • Common styles: Layered (with hardware abstraction layer), pipe-and-filter

5. Product-Line (Software Families) Architecture

A set of related software products built from a common set of shared assets (components, frameworks, architecture). Variations are built by composition and configuration, not rewriting.

  • Examples: Microsoft Office (Word, Excel, PowerPoint share UI, spell-check, file format components), Google Workspace, ERP systems (SAP, Oracle)
  • Key technique: Feature flags, plugin architecture, dependency injection for variation points
  • LMS example: Academic library variant vs. public library variant — different fine rules, membership types, reporting formats, but same core catalogue and circulation engine

6. Data-Intensive and Batch Systems

Process large volumes of data. Performance at scale, data integrity, and throughput are paramount.

  • Batch processing: Process large datasets periodically (overnight fine calculation for all overdue books)
  • Stream processing: Process data in real-time as it arrives (Apache Kafka, Spark Streaming)
  • Examples: Data warehouses, analytics pipelines, billing systems, bank reconciliation
  • Common styles: Pipe-and-filter, Lambda architecture (batch + stream), repository

Design Patterns — GoF and Beyond Applied to LMS

Definition: A design pattern is a reusable solution to a commonly occurring problem within a given context in software design. Patterns are not code — they are conceptual templates. (Gamma, Helm, Johnson, Vlissides — Gang of Four, 1994)

Pattern Template (PNAME)

  • Name: A memorable identifier for the pattern
  • Problem: The context and forces that motivate the pattern
  • Solution: The structure of elements and their relationships
  • Consequences: Trade-offs and results of applying the pattern

Category 1: Creational Patterns

Singleton — ensure one instance

// LMS: Single database connection pool, single configuration manager
class DatabasePool {
    private static instance: DatabasePool;
    private connections: Connection[];
    
    private constructor() { this.connections = []; }
    
    static getInstance(): DatabasePool {
        if (!DatabasePool.instance) {
            DatabasePool.instance = new DatabasePool();
        }
        return DatabasePool.instance;
    }
}
// Both BookRepository and MemberRepository use same pool
Singleton pitfall: Makes testing hard (global state). Prefer dependency injection where possible. Thread-safety must be explicitly handled.

Factory Method — delegate object creation

// LMS: Create different notification types without knowing concrete class
interface Notification { send(recipient, message): void }
class EmailNotification implements Notification { ... }
class SMSNotification implements Notification { ... }

class NotificationFactory {
    create(type: string): Notification {
        if (type === 'EMAIL') return new EmailNotification();
        if (type === 'SMS') return new SMSNotification();
        throw new Error('Unknown notification type');
    }
}
// Caller never knows which class is instantiated

Builder — construct complex objects step by step

// LMS: Build complex search query
const results = new BookQueryBuilder()
    .withTitle("python")
    .withCategory("PROGRAMMING")
    .availableOnly()
    .publishedAfter(2020)
    .sortByTitle()
    .limit(20)
    .build()
    .execute();

Category 2: Structural Patterns

Adapter — make incompatible interfaces work together

// LMS: Legacy book catalogue uses XML; new system expects JSON
interface IBookSource { getBooks(): Book[] }

class LegacyCatalogueAdapter implements IBookSource {
    private legacyCatalogue: LegacyXMLCatalogue;
    
    getBooks(): Book[] {
        const xmlData = this.legacyCatalogue.fetchBooksXML();
        return this.parseXMLToBooks(xmlData); // adapt XML → Book[]
    }
}

Facade — simplified interface to a complex subsystem

// LMS: LibraryFacade hides complexity of borrowing a book
class LibraryFacade {
    issueBook(memberId: string, isbn: string): IssueResult {
        const member = this.memberService.findById(memberId);
        this.memberService.validateEligibility(member);
        const book = this.bookService.findByISBN(isbn);
        this.bookService.validateAvailability(book);
        const loan = this.circulationService.createLoan(member, book);
        this.notificationService.sendIssueConfirmation(member, loan);
        this.bookService.decrementAvailability(book);
        return IssueResult.success(loan);
    }
    // Controller just calls facade — doesn't orchestrate 6 services
}

Decorator — add responsibilities dynamically

// LMS: Add logging and caching to BookRepository without changing it
class LoggingBookRepository implements IBookRepository {
    constructor(private wrapped: IBookRepository) {}
    findByISBN(isbn: string): Book {
        console.log(`Searching book: ${isbn}`);
        const result = this.wrapped.findByISBN(isbn);
        console.log(`Found: ${result?.title}`);
        return result;
    }
}

Category 3: Behavioural Patterns

Observer — notify multiple objects of state change

// LMS: When a book is returned, notify multiple systems
class BookReturnedEvent {
    book: Book; member: Member; loan: Loan;
}

class CirculationService {
    private observers: BookReturnObserver[] = [];
    
    registerObserver(obs: BookReturnObserver) { this.observers.push(obs); }
    
    returnBook(loanId: string): void {
        const loan = this.processReturn(loanId);
        // Notify all observers — FineCalculator, ReservationQueue, Analytics
        this.observers.forEach(obs => obs.onBookReturned(loan));
    }
}
// Adding a new system that cares about returns = just register a new observer

Strategy — encapsulate interchangeable algorithms

// LMS: Different fine calculation strategies for different member types
interface FineStrategy { calculate(daysOverdue: int): decimal }

class StudentFineStrategy implements FineStrategy {
    calculate(days: int): decimal { return Math.min(days * 1.0, 50); }
}
class FacultyFineStrategy implements FineStrategy {
    calculate(days: int): decimal { return 0; } // Faculty exempt
}

class FineCalculator {
    constructor(private strategy: FineStrategy) {}
    calculateFine(loan: Loan): decimal {
        return this.strategy.calculate(loan.getDaysOverdue());
    }
}
// Swap strategies without changing FineCalculator

Template Method — define skeleton, defer details to subclasses

// LMS: Report generation — same steps, different output formats
abstract class ReportGenerator {
    generateReport(): void {  // template method
        const data = this.fetchData();      // same for all
        const processed = this.process(data); // same for all
        this.renderOutput(processed);        // DIFFERENT per subclass
    }
    
    abstract renderOutput(data: ReportData): void; // defer to subclass
}

class PDFReportGenerator extends ReportGenerator {
    renderOutput(data: ReportData): void { /* render PDF */ }
}
class ExcelReportGenerator extends ReportGenerator {
    renderOutput(data: ReportData): void { /* render Excel */ }
}

User Interface Design Principles Shneiderman + Nielsen

UI design is a critical part of software design. A technically perfect system that users cannot or will not use has failed. UI design bridges the gap between the system and the human using it.

The 3 Golden Rules of Interface Design (Shneiderman)

1. Place the user in control

Users should feel they are in command of the interface. They initiate actions; the system responds. Provide undo, customisation, and flexible interaction modes.

2. Reduce the user's memory load

Don't make users remember information across screens. Use recognition over recall. Show options; don't make users remember them. Provide defaults.

3. Make the interface consistent

Same actions should produce same results. Same visual elements should mean the same thing. Users transfer learning within and across applications.

Shneiderman's 8 Golden Rules

#RuleLMS Application
1Strive for consistencyAll forms use the same layout: label above field, required fields marked with *, primary action button always bottom-right
2Enable frequent users to use shortcutsKeyboard shortcuts: Ctrl+N = new member, Ctrl+B = search book, F5 = refresh catalogue
3Offer informative feedbackAfter issuing a book: "Book 'Python Crash Course' issued to Ravi Kumar. Due: 14 Jan 2025." — not just "Success."
4Design dialogues to yield closureIssue book flow has clear beginning (scan member), middle (select book), end (confirmation screen with all details)
5Offer error prevention and simple error handlingISBN field validates format before submission; member lookup prevents issuing to expired members
6Permit easy reversal of actions"Undo last issue" button on confirmation screen (within 5 minutes). Librarian can cancel loan same day.
7Support internal locus of controlNever lock users in a modal they cannot escape. Always provide Cancel/Back. Don't auto-submit forms.
8Reduce short-term memory loadMember's name and photo displayed throughout loan process — no need to remember member ID across screens

Nielsen's 10 Usability Heuristics

1. Visibility of system status

Always keep user informed about what's happening. LMS: Progress bar for batch imports; loading spinner for search.

2. Match system and real world

Use words/concepts the user knows. LMS: "Issue" not "CREATE_LOAN_RECORD", "Return" not "TERMINATE_LOAN_SESSION".

3. User control and freedom

Undo and redo for mistakes. LMS: Cancel book issue; undo fine payment.

4. Consistency and standards

Follow platform conventions. LMS: Same date format throughout; same button styles for same actions.

5. Error prevention

Better than good error messages: design that prevents errors. LMS: Disable "Issue" button if member has unpaid fines.

6. Recognition over recall

Make options visible. LMS: Show list of recently searched books; autocomplete member names.

7. Flexibility and efficiency

Accelerators for experts. LMS: Barcode scanner for rapid book issue; bulk operations for librarians.

8. Aesthetic and minimalist design

Only show relevant info. LMS: Issue screen shows only fields needed for issuing — not member's full profile.

9. Help recognise, diagnose errors

Plain language errors. LMS: "This book has 0 copies available. 3 copies are currently on loan." not "Error: availability=0".

10. Help and documentation

Easy to search, focused on task. LMS: Context-sensitive help — clicking ? on fine screen shows fine policy.

Case Study: Library Management System (LMS) Full Design

System Overview

Client:City Public Library — 50,000 books, 5,000 active members
Users:Librarians (10), Members (self-service), Admin (2)
Key challenge:Replace paper-based system; integrate online member self-service portal
Architecture chosen:3-tier layered architecture (suitable scale, team of 5, monolithic deployment)
Genre:Open system (web-based) + Product-line (adaptable for other libraries)

Architecture Design Decision

Why Layered Architecture (not Microservices)? Team of 5 developers, single deployment, ~5,000 members. Microservices would add distributed system complexity with no benefit at this scale. Layered architecture gives clear SoC, easy testing, and familiar pattern for the team. Revisit for microservices if system scales to 50,000+ members across multiple branches.
