Testing, Deployment and Maintenance Unit 4

The science and practice of verifying software works correctly, deploying it safely, and keeping it running over time.

Why testing is not optional

Software without testing is a bet — not engineering. Testing does not prove the absence of defects, but it systematically reduces risk before software reaches users. The cost of finding a defect grows exponentially the later it is found.
Requirements phase defect

Cost to fix: $1. Found during requirements review in 10 minutes.

Design phase defect

Cost to fix: $10. Requires design rework before coding starts.

Coding phase defect

Cost to fix: $100. Code written to wrong spec must be rewritten.

Testing phase defect

Cost to fix: $1,000. Full regression testing required after fix.

Production defect

Cost to fix: $10,000+. Emergency fix, hotfix deployment, customer impact, reputation damage.

The testing landscape — complete picture

Acceptance TestingCustomer validates system meets their needs (Alpha, Beta, UAT)
System TestingComplete integrated system tested against all requirements (functional + NFR)
Integration TestingModule interfaces and interactions tested — find interface defects
Unit TestingIndividual functions/modules tested in isolation — find logic defects

Each level builds on the previous. Defects found at lower levels are cheapest. The pyramid: many unit tests → fewer integration → fewer system → fewest acceptance tests.

Testing process mnemonic: "OPVTRD"
Objectives → Plan → Verify/Design test cases → Test execution → Report → Debugging and fix

Testing Objectives and Principles Foundation

Software testing objectives

Primary objective: To find defects before the user does, with minimum effort and maximum coverage.
  • Find defects: Reveal errors in requirements, design, or code that could cause failures in production
  • Assess quality: Give stakeholders confidence in the software's fitness for purpose
  • Prevent defects: Early testing and reviews prevent defects from being injected downstream
  • Reduce risk: Identify and mitigate risks before deployment
  • Verify correctness: Confirm software behaves as specified
  • Validate fitness: Confirm software meets actual user needs (beyond the specification)
Dijkstra's Principle: "Testing can show the presence of bugs but never their absence." No amount of testing can prove a program is 100% correct — it can only increase confidence. This is why risk-based testing prioritises testing the most critical paths first.

Error vs Fault vs Failure — precise definitions

Error (Mistake)

A human action that produces an incorrect result. Developer misunderstands a requirement or makes a typo. Cause.

Fault (Bug/Defect)

A flaw in the software artifact (code, design, spec) resulting from an error. The bug sitting in the code, discovered by testing or code review. Manifestation of error.

Failure

Observable incorrect behaviour when the system executes. The fault activates under specific conditions and produces wrong output. Observable consequence.

Analogy: A developer types age > 18 instead of age >= 18 (error → fault in code). When a user who is exactly 18 tries to register, the system rejects them (failure). Not every fault causes a failure — some faults lie dormant until a specific input triggers them.

Verification vs Validation Exam Favourite

These two terms are among the most confused in software engineering. Both are essential. Both ask a different question.

The core distinction

Verification — "Are we building the product right?"

Check that each phase's output conforms to the specification from the previous phase. Process-oriented. Static activities are sufficient.

  • Checks against: SRS, design documents, standards
  • Methods: Reviews, inspections, walkthroughs, static analysis
  • Who: QA team, developers, analysts
  • When: During development, after each phase
Validation — "Are we building the right product?"

Check that the final system meets actual user needs and intended use. Product-oriented. Requires running the software.

  • Checks against: Customer's real needs (not just written spec)
  • Methods: Testing, prototyping, UAT, beta testing
  • Who: Customers, end users
  • When: End of development; each sprint in Agile

Classic failure — verified but not validated: Healthcare.gov (US, 2013) was technically built to specification (verified). The specification was wrong — it didn't reflect actual usage patterns. The system failed at launch (validation failure). You can pass every test and still ship the wrong product.
Both must succeed. A system that passes all functional tests but doesn't match user workflow has failed validation. A system that matches user needs but has hidden race conditions has failed verification. Strong engineering requires both.

Types of Testing — Levels Core Exam Topic

1. Unit Testing

Definition: Testing individual components (functions, methods, classes) in isolation from the rest of the system. The smallest testable unit is verified independently.
  • Who: Developers (written alongside code in TDD; written before in BDD)
  • Scope: One function, method, or class at a time
  • Tools: JUnit (Java), pytest (Python), Jest (JavaScript), NUnit (.NET)
  • Isolation: Use stubs and mocks to replace real dependencies
  • Key technique: White-box testing — tester knows the code
// Unit test example (pytest) — LMS fine calculation
def test_calculate_fine_one_day_overdue():
    loan = Loan(due_date=date(2024, 1, 1), return_date=date(2024, 1, 2))
    fine = FineCalculator().calculate(loan, MemberType.STUDENT)
    assert fine == 1.0  # ₹1/day for student

def test_calculate_fine_capped_at_maximum():
    loan = Loan(due_date=date(2024, 1, 1), return_date=date(2024, 6, 1))
    fine = FineCalculator().calculate(loan, MemberType.STUDENT)
    assert fine == 200.0  # capped at ₹200 max, not 150 * 1 = 150 days

def test_calculate_fine_within_grace_period():
    loan = Loan(due_date=date(2024, 1, 1), return_date=date(2024, 1, 2))
    # 1-day grace period — should be ₹0
    fine = FineCalculator().calculate(loan, MemberType.STUDENT)
    assert fine == 0.0

2. Integration Testing

Definition: Testing the interactions between integrated modules. Reveals defects in interfaces, data flow between modules, and combined behaviour.
  • Who: Development team / QA team
  • Finds: Interface mismatches, incorrect assumptions about module contracts, data format errors
  • Approaches: Big Bang, Top-Down, Bottom-Up, Sandwich/Hybrid

Stubs vs Drivers

Stub (used in Top-Down)

Replaces a called module (below) that doesn't exist yet. Returns canned responses so the caller can be tested.

Driver (used in Bottom-Up)

Replaces a calling module (above) that doesn't exist yet. Simulates calls to the module being tested.

3. System Testing

Definition: Testing the complete, integrated system against all specified requirements — both functional and non-functional. The system is treated as a black box.
  • Who: Independent QA team (not the developers)
  • Scope: End-to-end workflows; system as a whole
  • Includes: Functional testing, performance testing, security testing, usability testing, compatibility testing, recovery testing, stress testing

Alpha and Beta Testing Exam Topic

Alpha Testing

Definition: Testing performed by the development organisation itself (internal users, QA team, or selected internal staff) at the developer's site, before releasing to external users.
  • Who: Internal users, QA team, selected in-house staff — NOT the development team itself
  • Location: Developer's premises / controlled environment
  • Simulates: Real user behaviour but in a controlled setting
  • Precedes: Always done before Beta testing
  • Tools available: Testers can ask developers for clarification; developers can observe and log in real time
  • Goal: Find and fix major defects before exposing to external users
LMS Alpha testing example: Before launching the Library Management System to City Library staff, the software company's own employees use the system as if they were librarians and library members. They perform all workflows (register member, issue book, return, generate report). Developers observe and log issues. Major bugs are fixed before Beta release.

Beta Testing

Definition: Testing performed by a limited number of real external users at their own locations (customer site / real environment), without the development team present.
  • Who: Selected external users, paying customers, early adopters
  • Location: Customer's real environment — real hardware, real network, real workflows
  • Simulates: True production conditions the developer cannot fully replicate in a lab
  • Follows: Always done after Alpha testing
  • Developers: Not present; users submit bug reports independently
  • Goal: Find defects that only surface in the real environment; get user feedback on usability
LMS Beta testing example: The LMS software is deployed at 2 pilot libraries. Their librarians and members use it for 4 weeks in real operations. They submit bug reports, usability issues, and feature requests. The development team collects this feedback, prioritises fixes, and ships the production release.

White-Box Testing Code-Based Testing

Definition: White-box (glass-box / clear-box) testing is a technique where test cases are designed based on knowledge of the internal structure and logic of the software. The tester can "see inside" the code.

Objectives of white-box testing

  • Ensure every line of code is executed at least once
  • Test all decision branches (true and false)
  • Test all logical conditions
  • Test all loops (zero times, once, multiple times, boundary)
  • Find dead code (unreachable code)

Coverage criteria — types and hierarchy

1. Statement coverage (C0)

Every executable statement in the program must be executed at least once.

def calculate_discount(price, member_type):
    discount = 0                        # S1
    if member_type == "PREMIUM":        # S2
        discount = price * 0.20         # S3
    total = price - discount            # S4
    return total                        # S5

# One test case: calculate_discount(100, "PREMIUM")
# Covers: S1, S2 (True), S3, S4, S5 = 5/5 statements = 100% statement coverage
# BUT — what if member_type is NOT "PREMIUM"? S3 never tested via False path
Statement coverage weakness: 100% statement coverage does not mean all branches are tested. The False branch of an if-statement may never execute but statement coverage still reports 100%.

2. Branch / Decision coverage (C1)

Every decision (branch) must execute both True and False outcomes at least once.

def calculate_discount(price, member_type):
    discount = 0
    if member_type == "PREMIUM":  # Decision: must test True AND False
        discount = price * 0.20
    total = price - discount
    return total

# Test 1: calculate_discount(100, "PREMIUM")   → True branch (S3 executed)
# Test 2: calculate_discount(100, "STUDENT")   → False branch (S3 skipped)
# 2 test cases → 100% branch coverage

3. Condition coverage

Each individual boolean sub-expression (condition) must evaluate to both True and False at least once, independently.

if (age >= 18 AND income > 50000):    # 2 conditions: age>=18 AND income>50000
    approve_loan()

# Condition coverage requires:
# Test 1: age=20(T), income=60000(T)
# Test 2: age=16(F), income=60000(T)
# Test 3: age=20(T), income=30000(F)
# Branch coverage just needed T/F for the whole condition; 
# condition coverage independently tests each sub-condition

4. Path coverage (C2) — most thorough

Every possible path through the program must be executed. Impractical for complex programs (exponential paths with loops), but the gold standard for critical modules.

5. Cyclomatic Complexity (McCabe's metric)

Formula: V(G) = E − N + 2P = number of decision points + 1
  • E = edges in control flow graph, N = nodes, P = connected components
  • Simpler: count all if, while, for, case, and, or statements and add 1
  • V(G) = minimum number of test cases for full branch coverage
  • V(G) ≤ 10: manageable. V(G) 11–20: complex. V(G) > 20: very high risk — refactor
def issue_loan(member_id, isbn):
    member = find_member(member_id)  
    if not member:                       # decision 1
        return Error("Member not found")
    if not member.is_active():           # decision 2
        return Error("Member inactive")
    book = find_book(isbn)               
    if not book:                         # decision 3
        return Error("Book not found")
    if not book.is_available():          # decision 4
        return Error("Book unavailable")
    loan = create_loan(member, book)
    return loan

# V(G) = 4 decisions + 1 = 5
# Minimum 5 test cases needed for full branch coverage

Path coverageStrongest — every path. Impractical for loops. Use for critical modules.
Condition coverageEach boolean sub-expression True and False independently

Equivalence Partitioning (EP)

Divide input data into classes (partitions) where all values in a class are expected to behave identically. Test one representative value from each partition. Reduces the number of test cases needed.

Rules for identifying partitions

  • For a range: one valid partition (inside range), one below, one above
  • For a specific value: one valid (that value), one invalid (anything else)
  • For a set: one valid (member of set), one invalid (non-member)
  • Boolean: one true partition, one false partition

LMS example — member age for library card

Requirement: Member age must be between 5 and 100 (inclusive) for library membership.

Partitions:
  Invalid below:  age < 5        → test with age = 2
  Valid:          5 ≤ age ≤ 100  → test with age = 25
  Invalid above:  age > 100      → test with age = 105

3 partitions → 3 test cases (instead of testing ages 0-200)

Boundary Value Analysis (BVA)

Errors cluster at boundaries between partitions. BVA tests at the exact boundaries and values adjacent to them. Complements equivalence partitioning.

For a range [min, max], test: min−1, min, min+1, typical middle, max−1, max, max+1

Requirement: Fine amount must be between ₹0 and ₹200 (inclusive).
Min = 0, Max = 200

BVA test values:
  -1  → invalid (boundary below min)
   0  → valid (min boundary)
   1  → valid (just above min)
  100 → valid (typical)
  199 → valid (just below max)
  200 → valid (max boundary)
  201 → invalid (boundary above max)

7 test cases for complete boundary coverage

Decision Table Testing

Model complex business rules involving combinations of conditions. Each column is a test case (rule). Ensures all condition combinations are tested systematically.
LMS: Issue book decision table
Conditions (C):
  C1: Member active?
  C2: Member has unpaid fines > threshold?
  C3: Book available?

Actions (A):
  A1: Issue book
  A2: Block — member inactive

State Transition Testing

Design test cases based on the states a system can be in and the events that cause transitions. Tests all valid and invalid state transitions.
LMS: Loan state machine
States: ACTIVE → OVERDUE → RETURNED (also: ACTIVE → RETURNED)

Valid transitions:
  ACTIVE + "due date passes" → OVERDUE     (test: pass due date without return)
  ACTIVE + "return book"     → RETURNED    (test: return before due date)
  OVERDUE + "return book"    → RETURNED    (test: return after due date — fine applies)

Invalid transitions to test:
  RETURNED + "return book"   → should reject (already returned)
  RETURNED + "renew"         → should reject (cannot renew returned loan)

Use Case Testing

Design test cases from use case descriptions — main flow, alternate flows, and exception flows. Ensures end-to-end scenarios are tested as the user would experience them.

Use Case: Issue Book
Main flow test: Valid member scans card → scans book → confirms → success
Alt flow tests:
  Member has unpaid fines → system blocks and shows fine message
  Book has 0 copies → system shows "unavailable" with reservation option
Exception tests:
  Network fails mid-transaction → loan not created; book not decremented

Test Case Design Techniques Applied Practice

What makes a good test case?

A test case is a set of conditions under which a tester determines whether a system under test satisfies requirements or works correctly.

Required components of every test case:

  • Test case ID: Unique identifier for traceability (TC-001)
  • Test name: Short description of what is being tested
  • Pre-conditions: System state required before executing the test
  • Input data: Exact values to provide
  • Steps: Step-by-step actions to execute
  • Expected result: What should happen if the system is correct
  • Actual result: What actually happened (filled during execution)
  • Pass/Fail: Determined by comparing expected vs actual
  • Priority: Critical / High / Medium / Low
  • Traceability: Which requirement this test validates

Debugging strategies

1. Brute force / Print debugging

  • Insert print/log statements to trace execution and variable values
  • Simplest approach; clutters code; useful for quick investigation
  • Modern equivalent: structured logging with log levels (DEBUG, INFO, WARN, ERROR)
# Brute force debugging in LMS
def calculate_fine(loan, member_type):
    print(f"DEBUG: loan.dueDate={loan.due_date}, today={date.today()}")
    days_overdue = (date.today() - loan.due_date).days
    print(f"DEBUG: days_overdue={days_overdue}")
    rate = RATES[member_type]
    print(f"DEBUG: rate={rate}")
    fine = min(days_overdue * rate, MAX_FINE)
    print(f"DEBUG: fine={fine}")
    return fine

2. Backtracking

  • Start from the point of failure and work backwards through the code
  • Trace which state change caused the incorrect output
  • Effective for small programs; impractical for large codebases

3. Cause elimination (binary search / divide and conquer)

  • Progressively narrow down the location of the fault by eliminating sections of code
  • Comment out half the code; if the failure disappears, fault is in that half; repeat
  • Most systematic approach for large codebases
  • Git bisect automates this: binary searches commit history to find the commit that introduced a bug
# Git bisect — automated cause elimination
git bisect start
git bisect bad HEAD           # current commit is broken
git bisect good v1.2.0        # this commit was working
# Git automatically checks out midpoint commit
# Test: does bug exist? mark good or bad
git bisect good  # or: git bisect bad
# Repeat until git identifies the exact breaking commit

4. Induction / Deduction

Induction

Start from specific symptoms (test failures), identify patterns in the data, form a hypothesis, and test it. Specific → General.

Deduction

Start from general knowledge (known failure categories), deduce which type matches the symptoms, test that hypothesis. General → Specific.

5. Interactive debugging (using a debugger)

  • Breakpoints: Pause execution at a specific line to inspect state
  • Step over: Execute the current line and move to the next (don't enter function calls)
  • Step into: Enter the function being called on the current line
  • Step out: Complete current function and return to caller
  • Watch expressions: Monitor value of variables as execution progresses
  • Call stack: See sequence of function calls that led to current position

Debugging pitfalls and best practices

  • Fix the symptom, not the cause: Adding if fine < 0: fine = 0 when the real bug is in the days calculation — this hides the fault rather than fixing it
  • Introducing new bugs during fix: Always run full regression test suite after every fix
  • Not reproducing before fixing: "Works on my machine" — debugging without a reliable reproduction is guesswork
  • Fixing without understanding: Random change that accidentally fixes the symptom — leaves the root cause
Debugging mindset: The bug is not where you expect it to be — if it were, you'd have found it already. Question your assumptions about what the code does. Add assertions. Let the evidence lead the hypothesis, not the other way around.

Software Maintenance Post-deployment Life

Definition (IEEE): Software maintenance is the modification of a software product after delivery to correct faults, improve performance or other attributes, or adapt the product to a changed environment.
The maintenance reality: Software maintenance consumes 60–80% of total software lifecycle cost. More effort is spent maintaining existing software than building new systems. Understanding maintenance is as important as understanding development.

The 4 types of maintenance (Swanson, 1976)

1. Corrective Maintenance

Definition: Diagnosis and correction of errors (bugs/defects) discovered after delivery.

Trigger: Bug report from user or production monitoring alert

LMS example: Library manager reports that overdue fines are being calculated incorrectly for faculty members — showing $2/day instead of $0.50/day. Developer investigates, finds wrong rate constant used, fixes and redeploys.

Urgency: Depends on severity — P1 (system down) → immediate fix; P3 (cosmetic) → next scheduled release

~20% of maintenance effort (can be higher for immature systems)

2. Adaptive Maintenance

Definition: Modification of software to keep it usable in a changed or changing environment. The software itself is not broken — the world around it changed.

Triggers: OS upgrade, new database version, API change, regulatory change, hardware upgrade, cloud migration

LMS example: Python 3.8 end-of-life — LMS must be migrated to Python 3.12. Database vendor changes REST API — LMS integration must be updated. Government changes library data privacy law — LMS must add consent management.

~25% of maintenance effort

3. Perfective Maintenance

Definition: Enhancement of the software by adding new features, improving existing functionality, or improving performance/maintainability — based on user requests or strategic improvements.

Triggers: User feature requests, performance improvements, usability improvements, code quality improvements (refactoring)

LMS example: Library members request a mobile app. Admin wants analytics dashboard. Search is too slow — implement full-text search with PostgreSQL. Code is unmaintainable — refactor monolithic BookService into smaller services.

~50% of maintenance effort — the largest category

4. Preventive Maintenance

Definition: Activities to prevent future failures. Improve internal structure, quality, and reliability proactively — before problems manifest. Also called software re-engineering.

Triggers: Complexity metrics exceed threshold, high defect rate in a module, technical debt accumulation, security vulnerability audit

LMS example: Code review reveals LoanService has cyclomatic complexity >25. Developer refactors it into smaller, testable functions before it causes bugs. Security audit recommends updating password hashing from MD5 to bcrypt — done proactively, not after a breach.

~5% of maintenance effort but highest ROI per effort spent

How easy is it to understand the software to find the cause of a failure or identify what to change?

Software Configuration Management (SCM) Control the Change

Definition: SCM is the discipline of identifying, organising, and controlling modifications to software throughout the development and maintenance lifecycle. It answers: which version of what changed by whom, when, and why?

Why SCM is essential

  • Without SCM: "Which version is in production?" — nobody knows for certain
  • Multiple developers change the same file simultaneously — who wins?
  • A bug is fixed in one version but the fix never makes it to production — SCM prevents this
  • Regulatory compliance requires an audit trail of all changes
  • Emergency rollback: "Production is broken — revert to last working version immediately"

SCM core activities

1. Configuration identification

Identify all software artifacts that must be controlled — define configuration items (CIs).

  • Source code files, test cases, build scripts, deployment scripts
  • Requirements documents, design documents, SRS, architecture docs
  • Test plans, test results, defect reports
  • Third-party libraries and their versions
  • Environment configuration files (.env, docker-compose.yml)

2. Configuration control (Change management)

Formal process for requesting, evaluating, approving, and implementing changes to CIs after they are baselined.

Change Control Board (CCB): Cross-functional group (PM, architect, QA, business) that approves significant changes. Prevents uncontrolled scope creep and ensures impact is assessed before implementation.

3. Configuration status accounting

Record and report the status of all CIs and changes. Provides traceability: what version of each component is deployed where, which CRs have been implemented, and what is pending.

4. Configuration audits

  • Functional configuration audit (FCA): Verify the software's functional characteristics match the approved SRS
  • Physical configuration audit (PCA): Verify all items on the release manifest are actually present and correct

5. Version control

Track all changes to every file with full history — who changed what, when, and why.

6. Release and deployment management

  • Baseline: A formally agreed, reviewed configuration of a CI — snapshot used as the basis for further development. Once baselined, all changes must go through formal change control.
  • Release: A specific, named version of the software delivered to customers (v2.4.0, v2.4.1)
  • Deployment: Installing and configuring the release in the target environment (production, staging, DR)

Risk Management Proactive Risk Control

Definition: Risk management is the systematic process of identifying, analysing, prioritising, and responding to project risks to minimise the probability and impact of adverse events on a software project.

What is a software risk?

A risk is a potential problem that may occur. It has two key properties:

  • Probability (likelihood): How likely is it to occur? (0–1 or %)
  • Impact (consequence): If it occurs, how bad will it be? (Low/Medium/High/Critical)
Risk Exposure = Probability × Impact — prioritise risks with the highest exposure (most likely to happen AND most damaging).

Risk categories

Project risks

Affect schedule, budget, staffing, scope. Key developer leaves, budget cut, scope expansion, vendor delays.

Technical risks

Affect quality, architecture, implementation. Unproven technology, complex algorithms, performance unknowns, integration difficulties.

Business risks

Affect product viability. Competitor launches similar product, market shift, organisation changes strategy, regulatory change.

Known risks

Identified by project analysis — can be managed with RMMM plan.

Predictable risks

Based on historical data — past projects show these commonly occur.

Unpredictable risks

Emerge during the project — cannot be fully anticipated. Managed with contingency budget.

Risk identification
→

→

Step 1: Risk identification techniques

  • Risk checklists: Standard lists of common risks (staffing, requirements volatility, technology, etc.)
  • Brainstorming: Team generates risks; facilitator records all without judgment
  • SWOT analysis: Strengths, Weaknesses, Opportunities, Threats
  • Assumption analysis: Every project assumption is a potential risk if wrong
  • Delphi technique: Anonymous expert opinions gathered, iterated until consensus
  • Past project retrospectives: What went wrong before? What risks materialised?
