Introduction to Software Engineering Foundation

Unit 1 — All fundamentals from first principles to software crisis

What is Software?

Definition

Software = Programs + Data + Documentation. It is the set of instructions that controls hardware to produce a desired output.

Not just code

Software includes executable programs, configuration data, documentation, and supporting procedures.

Categories

  • System software (OS, compilers)
  • Application software
  • Embedded software
  • Web applications
  • AI/ML software
  • Scientific software

What is Software Engineering? Core Definition

IEEE Definition: "The application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software."

It is the application of engineering principles to software development to produce reliable, efficient, maintainable software within budget and schedule constraints.

Why Engineering? Writing code is easy. Building software that is correct, maintainable, scalable, and delivered on time/budget — that's engineering.

The Software Crisis High Exam Weight

Definition: The software crisis refers to the difficulty of writing useful, efficient, and maintainable computer programs in the late 1960s–1980s. It led to the formalization of Software Engineering as a discipline.

Symptoms of Software Crisis

Cost overruns

Projects costing 2–10x the original budget

Schedule slippage

Delivery months or years late

Poor reliability

Software that crashes, fails, has bugs in production

Unmet requirements

Software doesn't do what the customer wanted

Unmaintainable code

Adding features breaks existing functionality

No documentation

Knowledge trapped in developers' heads

Causes of Software Crisis

  • No methodology: Code-and-fix approach with no planning
  • Complexity: Software grew faster than the ability to manage it
  • No tools: Manual, error-prone development processes
  • Poor communication: Developers and clients didn't speak the same language
  • Unrealistic promises: Management underestimated difficulty
  • No metrics: No way to measure progress or quality

Solution: Birth of Software Engineering

The term "Software Engineering" was coined at the 1968 NATO Conference in Garmisch, Germany, as a deliberate response to the software crisis. The goal: apply engineering discipline to software development.

Evolution of Software Engineering Historical

1950s — Machine Era
Binary/Assembly code. One programmer, one machine. No methodology. "Clever hacking."
1960s — Software Crisis
High-level languages (FORTRAN, COBOL). Software complexity exploded. Crisis recognized. 1968 NATO Conference coins "Software Engineering."
1970s — Structured Era
Structured programming (Dijkstra). Waterfall model (Royce, 1970). First formal methodologies. GOTO-considered harmful.
1980s — OOP & Tools Era
Object-Oriented Programming (C++, Smalltalk). CASE tools. CMM model for process maturity. Prototyping introduced.
1990s — Process & Internet Era
UML (1994). RUP. RAD. Client-server & web development. "The Mythical Man-Month" revisited. Java emerges.
2000s — Agile Revolution
Agile Manifesto (2001). XP, Scrum, Kanban. Open-source explosion. SOA architecture. TDD, CI/CD emerge.
2010s–Now — DevOps & Cloud
DevOps, Microservices, Containers (Docker/Kubernetes), Cloud-native, AI-assisted coding. ML/AI software engineering challenges.

Characteristics of Good Software & Quality Attributes

Characteristics of Good Software Frequently Asked

Correctness

Software does exactly what the specifications say. Zero tolerance for wrong outputs.

Reliability

Performs its functions under stated conditions for a specified time. Measured as MTBF.

Efficiency

Uses minimal CPU, memory, bandwidth. Meets performance requirements under load.

Maintainability

Can be modified to fix bugs, improve performance, or adapt to new environment easily.

Usability

Easy to learn and use. Good UI/UX. Minimal user errors.

Portability

Can be transferred from one environment to another with minimal effort.

Scalability

Can handle increasing workload without performance degradation.

Security

Protected against unauthorized access, data breaches, attacks.

SDLC — Software Development Life Cycle Most Important

Definition: SDLC is a systematic process for planning, creating, testing, deploying, and maintaining software. It provides a structured sequence of stages for building software.

SDLC Phases

1. Requirement Analysis Gather, analyze, document requirements. SRS document produced. What should the system do?
2. System Design High-level architecture, database design, interface design. HLD and LLD documents.
3. Implementation (Coding) Actual code written based on design documents. Unit tests performed by developers.
4. Testing Integration, system, UAT. Defect finding and fixing. Verify system meets requirements.
5. Deployment Release to production. User training, rollout, go-live.
6. Maintenance Bug fixes (corrective), enhancements (perfective), adaptation (adaptive), prevention (preventive).

Types of Maintenance

TypePurposeExample
CorrectiveFix bugs discovered after deploymentFixing a login crash
AdaptiveAdapt to new environmentUpgrading for new OS
PerfectiveAdd new features, improve performanceAdding dark mode
PreventivePrevent future problems (refactoring)Code restructuring

Generic Process Model

A generic model captures the essence of all SDLC models. It defines a framework that any specific model (Waterfall, Spiral, etc.) instantiates.

  • All models must address: Requirements, Design, Implementation, Testing, Deployment
  • Variation is in the order, iteration, and overlap of these activities
  • Choice of model depends on: project size, requirement clarity, risk level, team size

Process Models Core Exam Topic

1. Waterfall Model Classic / Linear Sequential

Requirements

↓

↓

Advantages
  • Simple and easy to manage
  • Clear milestones and deliverables
  • Good documentation
  • Works well for small, well-defined projects
Disadvantages
  • No working software until end
  • Cannot accommodate changing requirements
  • Testing only at the end — late defect discovery
  • High risk for complex projects
Best for: Projects with clear, fixed requirements. Defense/government contracts. Small projects.
Worst for: Projects where requirements are unclear or will change.
Pitfall: The original Royce (1970) paper actually showed the Waterfall as FLAWED and advocated iterative development. Many people cite Waterfall without knowing this history.

2. Incremental Model Iterative Delivery

Software is built and delivered in increments (chunks). Each increment adds new functionality. Core product delivered first, then enhanced incrementally.

When to use: When you need early delivery of partial functionality. When requirements are moderately understood.
Risk: System architecture may not accommodate later increments well if not planned ahead.

Types: Staged delivery (increments pre-planned), Evolutionary (each increment refined based on feedback)

4. Spiral Model Most Complex — High Exam Weight

Boehm (1988): Risk-driven process model. Combines iterative development with the systematic control of the Waterfall model. Software is developed in a series of spirals (iterations).

Each spiral (loop) consists of 4 quadrants:

Quadrant 1: Planning Determine objectives, alternatives, constraints
Quadrant 2: Risk Analysis Identify, analyze, resolve risks. Prototype may be built here.
Quadrant 3: Engineering Design, code, test for this spiral's increment
Quadrant 4: Evaluation Customer evaluates progress; planning for next spiral
Advantages
  • Risk-driven — high-risk items addressed early
  • Suitable for large, high-risk projects
  • Incorporates user feedback every spiral
  • Flexible to requirement changes
Disadvantages
  • Expensive and complex to manage
  • Requires risk assessment expertise
  • Not suitable for small/low-risk projects
  • May go on forever (infinite spiral)
Key insight: In the Spiral model, risk is the primary driver. If you identify a risk early, you address it early. The number of spirals = number of iterations until the product is complete.

5. V-Model (Verification & Validation) Diagram Exam

Extension of Waterfall where each development phase has a corresponding testing phase. Called V-model because of the V-shape of the diagram.
Development (Left Side) ↔ Testing (Right Side)
Requirements Analysis⟺Acceptance Testing (UAT)
System Design⟺System Testing
Architectural Design⟺Integration Testing
Module Design⟺Unit Testing
▼ Implementation / Coding (Bottom of V) ▲
  • Verification: Are we building the product right? (Left side — review, inspection)
  • Validation: Are we building the right product? (Right side — testing)
  • Test planning happens simultaneously with development on the corresponding left phase
Advantage: Early test planning, higher chance of success. Clear correspondence between dev and test phases.
Disadvantage: Rigid. No iteration. No prototype. Late delivery. Not good for changing requirements.

  • Welcome changing requirements, even late in development

Scrum Framework All Details Required

Scrum is an agile framework for developing complex products using iterations called Sprints. Created by Ken Schwaber and Jeff Sutherland.

Scrum Roles

PO
Product Owner

Represents stakeholders. Owns the Product Backlog. Prioritizes features by business value. Says WHAT to build. Accepts or rejects sprint output.

Key tools: Product Backlog, User Stories, Release Plan

SM
Scrum Master

Facilitator and servant-leader. Removes impediments. Protects the team. Ensures Scrum is understood and followed. NOT a project manager.

Key activities: Facilitate ceremonies, remove blockers, coach team

DT
Development Team

Cross-functional, self-organizing team (3–9 members). Decides HOW to build. Owns the Sprint Backlog. Commits to Sprint goals.

Skills: Design, code, test — all in one team

Scrum Artifacts

Product Backlog

Ordered list of all features, enhancements, fixes needed. Owned by Product Owner. Never "done" — evolves continuously.

Contains: User Stories, Epics, Bugs, Tasks

Sprint Backlog

Subset of Product Backlog selected for the current Sprint. Team's plan for the Sprint. Owned by Dev Team.

Increment

Sum of all Product Backlog items completed during a Sprint. Must be in a useable, potentially shippable condition.

Case Study: Online Shopping System Applied Knowledge

System Overview

An Online Shopping System (e.g., Amazon-like) allows users to browse products, add to cart, checkout, and track orders. Sellers manage inventory. Admins manage platform.

Applying SDLC Phases

Phase 1: Requirement Analysis

Functional Requirements
  • User Registration & Login
  • Product Browsing & Search
  • Product Details Page
  • Shopping Cart Management
  • Order Placement & Checkout
  • Payment Processing
  • Order Tracking
  • Product Reviews & Ratings
  • Seller Product Management
  • Admin Dashboard
Non-Functional Requirements
  • Performance: Page load ≤2s
  • Scalability: 1M concurrent users
  • Availability: 99.99% uptime
  • Security: PCI-DSS compliance for payments
  • Usability: Mobile responsive
  • Reliability: Data consistency in cart
  • Maintainability: Microservices arch

Phase 2: System Design

High-Level Design (HLD)
  • 3-tier architecture: Client → Server → DB
  • Modules: Auth, Product, Cart, Order, Payment, Notification
  • Microservices: Each module = independent service
  • Database: PostgreSQL (orders), MongoDB (catalog), Redis (cart/cache)
  • CDN for static assets
Low-Level Design (LLD)
  • ER Diagram: Users, Products, Orders, Cart, Payments
  • API Design: RESTful endpoints
  • Class Diagrams for each module
  • Sequence diagrams for checkout flow

class User:
    def __init__(self, user_id, name, email):

class ShoppingCart:
    def __init__(self, user_id):
        self.user_id = user_id
        self.items = {}  # product_id: quantity

    def add_item(self, product_id, quantity=1):
        if product_id in self.items:
            self.items[product_id] += quantity
        else:
            self.items[product_id] = quantity

    def remove_item(self, product_id):
        if product_id in self.items:
            del self.items[product_id]

    def get_total(self, catalog):
        total = 0
        for pid, qty in self.items.items():
            total += catalog[pid].price * qty
        return total
